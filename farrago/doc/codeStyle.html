When In Rome, Do As the Romans Do

Every so often, all Java code will be automatically standardized via the Jalopy source code formatter. You can minimize the effects by following the example set by existing code. The Jalopy settings are defined in dev/farrago/src/jalopy.xml. (The most meaningful way to view this file is to import it into Jalopy's preferences GUI.)

In particular:

  • 80 column line length limit
  • all text files should be in Unix line-end format
  • set your editor to indent with spaces, not tabs
  • when creating a new file, please adapt the header from an existing nearby file

Naming conventions

Follow standard naming conventions:

package org.eigenbase.lowercase.packagename; 

public class CapitalizedClassName implements CapitalizedInterfaceName
{
    public static final int IntegerConstant = 0;
    int dataMemberName;

    void printOddWidgets(List widgetList)
    {
        int widgetCount = widgetList.size();
        for (int i = 0; i < widgetCount; i++) {
            if (i % 2 == 1) {
                Widget oddWidget = (Widget) widgetList.get(i);
                print(oddWidget);
            }
        }
    }
}

Short variable names such as "i" and "j" are acceptable for loop iterators, but otherwise use descriptive names. For cardinal variables, we prefer "widgetCount" to the Hungarian-style "nWidgets".

If a name contains an acronym such as "SQL", treat the acronym as if it were an ordinary word. Thus a class or interface would be called "SqlException" (not "SQLException"), variable or member name "sqlText" (not "SQLText" or "sQLText").
