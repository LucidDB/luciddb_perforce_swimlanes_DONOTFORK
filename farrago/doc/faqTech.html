Farrago Developer FAQ

How do I get a Java stack trace?

Some front-end tools allow you to see Java stack traces directly. For example, in sqlline, if you issue the !set verbose on command, a stack trace is reported with every subsequent error.

For other cases, you can often make use of the fact that all exceptions propagated through the Farrago JDBC driver are traced. In file dev/farrago/trace/FarragoTrace.properties, set net.sf.farrago.jdbc.level=FINER, then do whatever causes the exception and look at the tail of file dev/farrago/trace/FarragoTrace.log.

How do I get a Fennel C++ stack trace? How do I debug Fennel when it's called from Java?

At the moment, there's no nice automated stack trace like in Java (besides a standard core dump). However, it is possible to use gdb to debug Fennel. Before starting the Java virtual machine, set environment variable FENNEL_JNI_DEBUG=1 in the shell from which you are going to run Java. When the Fennel JNI layer is loaded, you should see a message like "Waiting for debugger; pid=18454". Start gdb and tell it to attach to the process with the given pid. The process should be sleeping in method JNI_OnLoad; set whatever breakpoints you want and then issue the gdb continue command (see below). Unfortunately, gdb receives many spurious signals when attached to the Java Virtual Machine (at least on Linux, even SIGSEGV!) The only choices are to tell gdb to ignore them (e.g. with the command handle SIGSEGV ignore), or to manually continue past them each time (like if you want to track down a real segfault).

On some systems (e.g., Fedora Core 2, but it probably depends on the version of gdb), the gdb continue command will not cause the call to sleep() to end prematurely. In this case, set FENNEL_JNI_DEBUG=2 and use the gdb command signal SIGHUP or signal 1 to end the call to pause() in JNI_OnLoad.

How do I trace the Fennel C++ unit tests?

In directory fennel/test, create a file named trace.cfg containing a single line: 

testTraceLevel 300
When you run the test you want to trace, specify this file as an argument to the executable, e.g. 

$ TupleTest trace.cfg
Running 4 test cases...

*** No errors detected

$ head TupleTest_trace.log
[TupleTest] ENTER:  testStandardTypesNotNull
...
By default, the output goes to a file based on the test name as shown above. There are other test parameters (TODO: document) besides testTraceLevel; e.g. testTraceFileName controls where the trace file is written.

The level 300 corresponds to FINEST in java.util.logging. To use a different level, see the TraceTarget enum in fennel/common/TraceTarget.h. There is not currently a way to specify different trace levels for different components (as is possible when tracing from Java).

To control the verbosity of Boost test framework messages, set environment variable BOOST_TEST_LOG_LEVEL.

How do I deal with bad Java code generated by the optimizer?

The first thing to do when you an encounter an internal error resulting from such a bug is to make sure you can see the code in question. In file FarragoTrace.properties, set net.sf.farrago.dynamic.level=FINE before executing the SQL statement. This will guarantee that the generated .java file will be preserved under directory dev/farrago/classes/net/sf/farrago/dynamic/stmtXYZ (where XYZ is a random unique suffix which you may be able to see in the error message). Now, use the compilation error together with the generated source code to find the problem (at this point, someone still has to debug the code generator to track down the origin of the problem and fix it). When you're done, turn off the trace (since the generated code clutters up disk space).

If the generated Java code runs but does the wrong thing, you may have to debug it if you can't find the problem by inspection. Set a breakpoint in the finally clause of method net.sf.farrago.db.FarragoDatabase.prepareStmt and execute the SQL statement. Once this breakpoint is reached, the dynamically compiled class should already be loaded. Now, find the place in your debugger where you can browse loaded classes (assuming there is one) and navigate to package net.sf.farrago.dynamic. If there's only one class listed under there, that's it. Otherwise, you can use the field packageName from the local variable stmt in the prepareStmt() method to find your class in the haystack. Now, set breakpoints in the generated class and off you go.

Why do my trace setting changes sometimes take effect immediately, but other times they're ignored until the next virtual machine invocation?

The trace system has a thread which polls file FarragoTrace.properties every second for changes. When changes are detected, the file is reloaded and the new settings are traced to FarragoTrace.log. This can be very useful in the middle of a debugging session.

However, some subsystems (Fennel in particular) cache trace settings for performance. Cached trace settings are typically associated with an object lifetime. For example, when a TupleStream graph is first prepared, extra trace streams are instrumented into the graph if the corresponding trace setting is set. These continue to trace for the lifetime of the graph. Conversely, if the graph was prepared when trace was off, no extra streams are created, and the graph's execution will never be traced even if the trace settings change in the middle. This is intentional, but when combined with object caching, it can make debugging mysterious. If you want to be sure of getting something traced, start a new VM after changing trace settings.

Why do some breakpoints get skipped entirely, even though they must have been hit in order for a statement to execute?

Farrago employs a fairly advanced cache strategy for objects used in the implementation of SQL statements. This means that after the first preparation request, subsequent preparation requests will be able to use the cached implementation, skipping most of preparation (and your breakpoints) entirely. As usual, this is good for performance but makes debugging more difficult. You can minimize implementation caching by executing the SQL statement alter system set "codeCacheMaxBytes"=min. When done, use alter system set "codeCacheMaxBytes"=max to remove the limit (otherwise normal operation will be much slower).

Why is it taking so long to run the tests? It used to be faster!

There's a good chance you forgot to turn off an unnecessary trace after a debugging session. Detailed tracing (such as Fennel) slows down the system significantly because each trace entry is flushed to disk individually (guaranteeing that it will be visible after a crash).

JSwat is complaining about UnknownHostException or Socket transport error; what do I do?

To solve the UnknownHostException, add the hostname of your computer to /etc/hosts, e.g.
127.0.0.1 localhost
192.168.0.2 myHostName
On Red Hat, run (as root) service network restart and try running jswat again.

If you get a Socket transport failed to init error when running jswat, make sure that the hostname you specified in /etc/hosts maps to the right ip address. Use ifconfig to get your ip address.

All the Farrago tests pass except the FarragoServerTest, what may be wrong?

Check that the IP address assigned to your hostname is correct in etc/hosts. Use ifconfig to get your ip address.
